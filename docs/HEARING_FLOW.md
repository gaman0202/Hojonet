# ヒアリングフォームの流れ（専門家送信 → ユーザー回答）

## 現在の構造

### 1. ユーザー（顧客）がヒアリングを見る場所

| 画面 | パス | 内容 |
|------|------|------|
| **案件詳細** | `/dashboard/cases/[id]`（例: `/dashboard/cases/13`） | タブのうち **「ヒアリング」** を選択するとヒアリングフォームが表示される。 |

- **表示データ**: 現状は **フロントの初期 state（ハードコード）** で、質問・テンプレート名が固定。
- **保存・提出**: 「提出する」で `PUT /api/cases/[caseId]/hearing` が呼ばれ、回答が `case_hearing_submissions` に保存される。
- データ未取得時は `HearingTab` 内で「ヒアリングフォームがありません。担当者からヒアリングフォームが届くとこちらに表示されます」と表示されるが、**初回表示時は上記の固定フォームが入った state のため、常に何かしらフォームが表示されている**。

### 2. 専門家がヒアリングを「送信」したとき

| 画面 | 操作 | API |
|------|------|-----|
| **案件詳細（専門家）** | `/expert/management/[id]/[caseId]` → 「追加ヒアリング」→ テンプレート選択・メッセージ入力 → **送信する** | `POST /api/expert/cases/[caseId]/hearing-request` |

- **API の動き**: `templateId` と `message` を受け取り、`activity_logs` に「ヒアリングフォームを送信した」旨のログを入れる（テーブルが無い・スキーマ違いの場合はスキップ）。
- **ユーザー側とのつながり**: 現状は **なし**。専門家が選んだテンプレートや質問一覧はユーザーのダッシュボードには渡っていない。

### 3. まとめ（現状のギャップ）

- **ユーザーがヒアリングを見る場所**: **`/dashboard/cases/[id]` の「ヒアリング」タブのみ。**
- **専門家が送信したテンプレート**は、現状どこにも保存・表示されていない。
- ユーザーが回答するフォームは **常に同じハードコードの質問** で、専門家が「送信」したフォームとは **別物**。

---

## 専門家送信 → ユーザー表示を繋ぐには（今後の拡張案）

1. **案件に「送信されたテンプレート」を紐づける**  
   - 例: `cases.hearing_template_id` や「ヒアリング依頼」用テーブルに `case_id` + `template_id` + 送信日時を保存。

2. **ユーザー側でヒアリングデータを API から取得**  
   - 例: 案件詳細のマウント時や「ヒアリング」タブ表示時に  
     `GET /api/cases/[caseId]/hearing`（または専用の「表示用ヒアリング取得」API）で  
     - テンプレート名  
     - 質問一覧（と選択肢）  
     - 既存の回答・下書き  
     を取得し、`hearingData` の state をそれで初期化する。

3. **専門家送信 API でテンプレートを保存**  
   - `POST /api/expert/cases/[caseId]/hearing-request` 内で、  
     送信された `templateId` を案件または「ヒアリング依頼」に保存し、  
     上記の「表示用ヒアリング取得」API がそのテンプレートを返すようにする。

以上を実装すると、「専門家が送信 → ユーザーは `/dashboard/cases/[id]` のヒアリングタブでそのフォームを見て回答」という一連の流れが繋がります。
