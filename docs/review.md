# プロジェクト同期／ステータス更新の概要

## 概要

現在のプロトタイプは基本機能（ダッシュボード、案件管理、メッセージ、テンプレート等）が概ね動作する一方、フォームのバリデーション不具合、ナビゲーションの戻り動線、顧客中心の情報表示、書類アップロード運用（メール依存）などに改善余地が多数ある。方針として、メールベースからアプリ内アップロード・メッセージ中心へ移行し、ダッシュボード初期表示と顧客軸のナビゲーション強化を優先する。

## ユーザー側フローとUI改善

- マイダッシュボード
    - 「進行中の案件」「期限が迫っているタスク」が同一ページ遷移。緊急タスクは進行中内で強調表示（例：赤表示）する設計に整合。
    - 画面に戻るための矢印（バック）ボタンを設置し、誤操作防止と明確な戻り動線を提供。
- 「相談する」導線
    - ボタン位置が端で認知しづらい。画面中央～右上の目立つ位置に増設し、CTAを強化。
    - 相談開始後の情報入力フローは表示自体は出るが、バリデーション周りに不整合あり。
- 入力フォームとバリデーション
    - 必須項目未入力でも提出ボタンが活性化する不具合。必須入力完了まで非活性を維持する制御に修正。
    - 「プロフィールが見つかりません」エラーが発生。担当者プロフィールの存在チェックと紐付け処理の見直しが必要。
    - 数値入力欄は数字のみ許容。プラス/マイナス操作UIの仕様を明確化。
    - 一時保存が機能せず、データ保持に失敗。ドラフト保存APIとフロントの状態管理を再確認。
- 必須情報項目
    - 代表者名・担当者名の項目を追加し、少なくともどちらか必須とする。ただし担当者がいる場合は担当者名を必須化。
    - 連絡先・メールアドレスはアカウント登録情報から参照共有。申請者プロフィール閲覧可能とする。

## 専門家側フローと案件管理

- ダッシュボードと案件検索/管理
    - 「進行中の案件」タイルやタブが非クリック。全体をクリック可能化し操作性を改善。
    - 「案件検索」は公開案件から選択して「参加する」→「案件管理」に登録される流れを確認。
- ステータス管理UI
    - 横軸ステージ（ヒアリング/資料準備等）が横に長く、スクロール対応に課題。タブ全体クリックやページ内スクロールバーで可用性を確保。
    - 色分けルール：紫=進行中、黄緑=終了、グレー=未着手。ステージ移動は双方向に可能。
- 顧客別ビュー
    - 顧客ごとにタスク・提出書類を一覧し、専門家が目視確認後に許可/棄却を操作。
    - 顧客情報（会社名・連絡先）を画面上部に常時表示し、補助金情報よりも顧客識別を優先。

## 書類アップロード/処理と通知

- 現状と課題
    - n8nでメール受信→システム投入の運用は煩雑で誤認識リスクあり（メール本文の余計な記述等）。
- 方針と機能要件
    - アプリ内で直接アップロード（ドラッグ&ドロップ対応）を標準化し、アップロード通知を自動送信。
    - 新着/未確認書類に強調表示（例：NEWタグ）を付与して見落とし防止。
    - システム内でアップロード日付の妥当性チェック（例：3ヶ月以上前は不可）をOCRベースで自動判定する拡張を検討。
    - 顧客への連絡はメールではなくメッセージ機能で自動・一元管理。過去はメール返信だった内容をメッセージに集約。

## 顧客管理/ナビゲーション

- 顧客中心の情報アクセス
    - 顧客管理から該当補助金・案件へ遷移可能に。顧客ビューに「現在の進捗」「最終更新日」を表示し状況把握を容易化。
    - 案件詳細からの戻り動線は、「案件リストに戻る」「顧客情報に戻る」の2系統ボタンを設置し、戻り先の一貫性を担保。
- 表示情報の最適化
    - 顧客管理側では支援金額よりも進捗/更新情報の優先表示が有用。
    - 案件管理の補助金名表示に加え、担当顧客名の明示と顧客からの直接遷移を強化。

## テンプレート/メッセージ機能

- テンプレート
    - メッセージ/書類/タスク/ヒアリングフォームごとにテンプレート作成・分類が可能（書類、打ち合わせ、確認等）。
    - 優先度はテンプレートではなく顧客・案件に紐付ける概念が適切。テンプレは初期は空で、行政書士が随時作成。
- ヒアリングフォーム
    - テキスト/長文/ドロップダウン/チェックボックス/ラジオなどGoogleフォームに類似のUIで質問を追加・必須指定が可能。
- メッセージ
    - アプリ内で顧客とのコミュニケーションを完結可能。テンプレートからの送信や案件画面からのアクセスに対応。

## バックエンド/統合と運用

- フローの不明点
    - ログイン後に送信された情報がどこに格納され、どの画面に反映されるかの詳細が未確定。バックエンド実装・仕様整理が必要。
- n8n連携
    - 直接アップロード化に合わせ、n8n側は通知・ワークフロー調整に限定。アップロード通知のトリガー設定を再構成。
- 紹介者管理
    - 紹介者→管理顧客への紐付け表示を予定。現状サンプルデータなしで検証不可。

## 合意事項/方針

- メール依存からアプリ内アップロード＋メッセージ運用へ移行し、通知は自動化。
- ダッシュボードを初期表示にし、未使用の「補助金検索」画面等は当面非表示化。
- 一般ユーザーに「専門家」ボタンは不要。専門家は専用の登録フォームから導線提供（現仕様の意図は確認する）。
- 顧客軸のナビゲーションを強化し、進捗/更新情報の可視化を優先。
- フロント側の入力チェック・UI改善は優先課題。バックエンドの格納先/遷移仕様は別途整理。
- 他の行政書士によるユーザビリティ確認を実施し、初見での操作性を検証。

## リスク/ブロッカー

- 登録不可・一時保存不可などの致命的バグでフロー検証が進まない。
- 担当者プロフィール未登録時のエラー処理不備。
- 戻り動線の不統一によるユーザー混乱。
- 「専門家」ボタンの仕様意図が不明瞭で、誤操作の可能性。
- 紹介者機能の検証データ不足。

## 対応事項

- [o] フロント：提出ボタンの活性制御を必須項目完了まで非活性に修正（バリデーション全体の再点検）。
  > `/register` `/login` `/forgot-password` `/update-password` `/dashboard/profile/edit` `/expert/settings` `/admin/customers` の各ボタンに必須項目チェックを追加。`/hearing/[id]` のヒアリングフォームはprogress<70の曖昧な閾値をprogress<100（全項目入力必須）に修正。

- [o] フロント：矢印バックボタンを各主要ページに実装し、戻り先の一貫性を担保。
  > `/expert/management/[id]/[caseId]` のヘッダーに「案件リストに戻る」（`/expert/management/[id]`）と「顧客情報に戻る」（`/expert/customers/[customerGroupId]`）の2系統ボタンを追加。`/dashboard/cases/[id]` は既存の「案件一覧に戻る」が正常実装済みのため対応不要。

- [保留] フロント：相談するボタンの配置を画面中央付近に増設し、CTAの視認性を向上。

- [o] フロント/バック：代表者名・担当者名フィールドを追加し、担当者名を必須化（いずれか必須のロジック実装）。
  > `/register` と `/dashboard/profile/edit` のフォームに「代表者名」「担当者名」フィールドを追加。どちらか一方が未入力の場合は送信ボタンを非活性に制御。`profiles` テーブルに `representative_name` / `contact_name` カラムを追加し、登録トリガー（`handle_new_user`）とプロフィール更新APIも対応。

- [o] バック：担当者プロフィール未登録時のエラーハンドリングと登録導線を整備。
  > `/api/hearing/submit` でプロフィールが存在しない場合、Auth メタデータから基本情報を自動生成（upsert）してフローを継続するよう修正。自動生成も失敗した場合は `error_code: profile_missing` を返し、`/hearing/[id]` 側でアンバー色のエラーバナーと「プロフィールを登録する」ボタンを表示。ボタンタップで `/dashboard/profile/edit?redirect=/hearing/[id]` へ遷移し、プロフィール保存後は元のヒアリングページに自動で戻るよう対応。

- [o] バック：ドラフト一時保存のAPI・フロント状態管理を修復し、保存成功を保証。
  > `/hearing/[id]` の「一時保存」ボタンに `handleSaveDraft` を実装。localStorageキー `hearing_draft_[id]` に入力内容を保存。ページ再訪問時に自動復元し「前回の入力内容を復元しました」バナーを4秒表示。保存成功時は「一時保存しました」グリーントーストを3秒表示。提出完了後はdraftを自動削除。

- [o] フロント：案件タイル/タブを全面クリック可能にし、操作性を改善。
  > `/expert/dashboard` の統計カード（進行中の案件→`/expert/management`、緊急タスク→`/expert/management`、未読メッセージ→`/expert/management`、登録済み補助金→`/expert/subsidies`）をクリック可能に変更。今週の予定の各項目も `/expert/management/[subsidyId]/[caseId]` へ直接遷移するよう対応。

- [o] フロント：案件詳細から「案件リストに戻る」「顧客情報に戻る」の2系統の戻りボタンを追加。
  > `/expert/management/[id]/[caseId]` のヘッダーに「案件リストに戻る」（`/expert/management/[id]`）と「顧客情報に戻る」（`/expert/customers/[customerGroupId]`）の2つのボタンを追加。`/dashboard/cases/[id]` は既存の「案件一覧に戻る」が正常に実装済みのため対応不要。

- [o] フロント：顧客ビューに「現在の進捗」「最終更新日」「会社名・連絡先」を常時表示。
  > `/expert/customers/[id]` のヘッダー右上に「進行中 N件」バッジと「最終更新: YYYY/MM/DD」を追加表示。`/expert/customers/[id]` のAPIレスポンスに `updatedAt` フィールドを追加。会社名・連絡先は既存表示済み。

- [o] バック/インテグレーション：アプリ内ドラッグ&ドロップアップロード機能を実装し、新着強調表示・アップロード通知をn8nで連携。
  > `/dashboard/cases/[id]` 書類タブの未提出・差戻しカードにドラッグ&ドロップ領域を追加。`/expert/management/[id]/[caseId]` の書類追加モーダルにも同様に適用。顧客が書類・タスクをアップロード・提出すると専門家画面にNEWバッジ（青）が表示され、専門家が確認・承認・差戻しすると自動で消える。逆に専門家がタスク追加・承認・却下した際は顧客画面にもNEWバッジを表示。通知はすでに毎朝9時に3日以内期限タスク・未読メッセージをメール（Resend）・LINEで自動送信する仕組みを実装済み。

- [保留] バック：OCRによるアップロード書類の日付妥当性チェックの設計・プロトタイプ作成。
  > 実装上の課題が多く優先度を下げて保留。主な問題：①日本語書類はOCR精度が低く、縦書き・印鑑・手書き混在で誤認識リスクが高い。②書類種別ごとに発行日の記載位置が異なり「どの日付が有効期限の基準か」の判定が困難。③PDFにテキストレイヤーがある場合はOCR不要だが、スキャン画像の場合はGoogle Vision API等の外部サービス連携が必要でコストが発生。④誤検知で有効な書類を弾いた場合のUX悪化リスクあり。行政書士が目視確認する現運用で十分カバーできるため、当面は対応を見送る。

- [o] プロダクト：初期表示をダッシュボードに変更し、当面「補助金検索」等は非表示化。
  > ホーム画面の「補助金目録・検索」カード文言を「補助金目録」に変更し、`/subsidies` 画面のテキスト検索ボックスは JSX コメント化して検索機能を一時的に無効化（一覧表示＋絞り込みフィルタのみ提供）。

- [o] プロダクト：一般ユーザーの「専門家」ボタンを非表示にし、専門家専用登録導線を設計（現仕様の意図を関係者へ確認）。
  > 現状、トップページ(`/`)ヘッダー右上に表示している「専門家／ユーザー／管理者」切り替えボタンは、あくまで開発・デモ用の暫定UIであり、本番想定の導線ではない。

- [ ] QA：専門家用アカウントでユーザーフロー～案件登録～書類処理までのE2Eテストを実施。

- [ ] UXR：他の行政書士にレビュー依頼し、初見操作性のフィードバックを収集。

- [ ] ドキュメント：本議事録の文字起こしと各修正点リストを事務所へ共有。

# 프로젝트 동기화 / 상태 업데이트 개요

## 개요

현재 프로토타입은 기본 기능(대시보드, 사건 관리, 메시지, 템플릿 등)이 대체로 동작하는 반면, 폼 유효성 검사 오류, 내비게이션 뒤로가기 동선, 고객 중심의 정보 표시, 서류 업로드 운영(이메일 의존) 등에 개선 여지가 많다. 방침으로서 이메일 기반에서 앱 내 업로드·메시지 중심으로 전환하고, 대시보드 초기 표시와 고객 축 내비게이션 강화를 우선시한다.

## 사용자 측 플로우 및 UI 개선

- 마이 대시보드
    - 「진행 중인 사건」「기한이 임박한 태스크」가 동일 페이지로 이동. 긴급 태스크는 진행 중 항목 내에서 강조 표시(예: 빨간색 표시)하는 설계로 통일.
    - 화면으로 돌아가기 위한 화살표(뒤로가기) 버튼을 설치하여 오조작 방지 및 명확한 뒤로가기 동선 제공.
- 「상담하기」 동선
    - 버튼 위치가 끝에 있어 인지하기 어려움. 화면 중앙~우측 상단의 눈에 띄는 위치에 추가 배치하여 CTA 강화.
    - 상담 시작 후 정보 입력 플로우는 표시 자체는 나타나지만, 유효성 검사 주변에 불일치 존재.
- 입력 폼과 유효성 검사
    - 필수 항목 미입력 상태에서도 제출 버튼이 활성화되는 버그. 필수 입력 완료까지 비활성 유지하는 제어로 수정.
    - 「프로필을 찾을 수 없습니다」 오류 발생. 담당자 프로필 존재 여부 확인 및 연결 처리 재검토 필요.
    - 숫자 입력란은 숫자만 허용. 플러스/마이너스 조작 UI 사양 명확화.
    - 임시 저장이 동작하지 않아 데이터 보존 실패. 초안 저장 API와 프론트 상태 관리 재확인.
- 필수 정보 항목
    - 대표자명·담당자명 항목을 추가하고, 최소 어느 쪽이든 필수로 지정. 단, 담당자가 있는 경우 담당자명을 필수화.
    - 연락처·이메일 주소는 계정 등록 정보에서 참조 공유. 신청자 프로필 열람 가능하게 설정.

## 전문가 측 플로우 및 사건 관리

- 대시보드 및 사건 검색/관리
    - 「진행 중인 사건」 타일이나 탭이 클릭 불가. 전체를 클릭 가능하게 하여 조작성 개선.
    - 「사건 검색」은 공개 사건에서 선택하여 「참가하기」 → 「사건 관리」에 등록되는 흐름 확인.
- 상태 관리 UI
    - 가로 축 스테이지(히어링/자료 준비 등)가 가로로 길어 스크롤 대응에 과제. 탭 전체 클릭이나 페이지 내 스크롤바로 가용성 확보.
    - 색상 구분 규칙: 보라색=진행 중, 황록색=완료, 회색=미착수. 스테이지 이동은 양방향 가능.
- 고객별 뷰
    - 고객별로 태스크·제출 서류를 목록화하고, 전문가가 육안 확인 후 허가/반려 조작.
    - 고객 정보(회사명·연락처)를 화면 상단에 항상 표시하여, 보조금 정보보다 고객 식별을 우선.

## 서류 업로드/처리 및 알림

- 현황과 과제
    - n8n으로 이메일 수신 → 시스템 투입 운영은 번거롭고 오인식 위험 존재(이메일 본문의 불필요한 내용 등).
- 방침 및 기능 요건
    - 앱 내에서 직접 업로드(드래그 앤 드롭 지원)를 표준화하고, 업로드 알림을 자동 발송.
    - 신규/미확인 서류에 강조 표시(예: NEW 태그)를 부여하여 누락 방지.
    - 시스템 내에서 업로드 날짜 유효성 검사(예: 3개월 이상 전은 불가)를 OCR 기반으로 자동 판정하는 확장 검토.
    - 고객 연락은 이메일이 아닌 메시지 기능으로 자동·일원 관리. 기존에 이메일 답장으로 처리하던 내용을 메시지로 집약.

## 고객 관리/내비게이션

- 고객 중심의 정보 접근
    - 고객 관리에서 해당 보조금·사건으로 이동 가능하게 설정. 고객 뷰에 「현재 진행 상황」「최종 업데이트일」을 표시하여 상황 파악 용이화.
    - 사건 상세에서의 뒤로가기 동선은 「사건 목록으로 돌아가기」「고객 정보로 돌아가기」의 2계통 버튼을 설치하여 뒤로가기 목적지의 일관성 담보.
- 표시 정보 최적화
    - 고객 관리 측에서는 지원 금액보다 진행 상황/업데이트 정보의 우선 표시가 유용.
    - 사건 관리의 보조금명 표시에 더하여, 담당 고객명 명시 및 고객으로부터의 직접 이동 강화.

## 템플릿/메시지 기능

- 템플릿
    - 메시지/서류/태스크/히어링폼별로 템플릿 작성·분류 가능(서류, 회의, 확인 등).
    - 우선순위는 템플릿이 아닌 고객·사건에 연결하는 개념이 적절. 템플릿은 초기에는 비어 있으며, 행정서사가 수시로 작성.
- 히어링 폼
    - 텍스트/장문/드롭다운/체크박스/라디오 등 Google 폼과 유사한 UI로 질문 추가·필수 지정 가능.
- 메시지
    - 앱 내에서 고객과의 커뮤니케이션 완결 가능. 템플릿으로부터의 발송이나 사건 화면에서의 접근 지원.

## 백엔드/통합 및 운영

- 플로우의 불명확한 점
    - 로그인 후 전송된 정보가 어디에 저장되고 어느 화면에 반영되는지의 상세가 미확정. 백엔드 구현·사양 정리 필요.
- n8n 연동
    - 직접 업로드화에 맞추어 n8n 측은 알림·워크플로 조정으로 한정. 업로드 알림 트리거 설정 재구성.
- 소개자 관리
    - 소개자 → 관리 고객으로의 연결 표시 예정. 현재 샘플 데이터 없어 검증 불가.

## 합의 사항/방침

- 이메일 의존에서 앱 내 업로드 + 메시지 운영으로 전환하고, 알림은 자동화.
- 대시보드를 초기 표시로 설정하고, 미사용 「보조금 검색」 화면 등은 당분간 비표시화.
- 일반 사용자에게 「전문가」 버튼은 불필요. 전문가는 전용 등록 폼에서 동선 제공(현 사양의 의도는 확인).
- 고객 축 내비게이션을 강화하고, 진행 상황/업데이트 정보의 가시화를 우선.
- 프론트 측 입력 체크·UI 개선은 우선 과제. 백엔드 저장 위치/이동 사양은 별도 정리.
- 다른 행정서사에 의한 사용성 확인을 실시하고, 초견 시 조작성 검증.

## 리스크/블로커

- 등록 불가·임시 저장 불가 등의 치명적 버그로 플로우 검증이 진행되지 않음.
- 담당자 프로필 미등록 시 오류 처리 미비.
- 뒤로가기 동선 불통일로 인한 사용자 혼란.
- 「전문가」 버튼의 사양 의도가 불명확하여 오조작 가능성.
- 소개자 기능의 검증 데이터 부족.
